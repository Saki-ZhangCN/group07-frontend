# 第一部分 引言

## 一、编写目的

编写本文的主要目的是把需求分析得到的用例模型转换为软件结构和数据结构。设计软件结构的具体任务是：将一个复杂系统按功能进行模块划分、建立模块的层次结构及调用关系、确定模块间的接口及人机界面等。数据结构设计包括数据特征的描述、确定数据的结构特性、以及数据库的设计。目的是为了准确阐述在线教育智慧教学平台的具体实现思路和方法

本设计是指导详细设计和项目实施的重要指导性文件，也是后续详细设计、项目实施、系统集成测试的核心指导文件以及系统集成测试的重要依据。



## 二、项目背景

当前，教育信息化2\.0行动持续推进，但传统在线教学平台仍面临诸多痛点：教学过程数据割裂，难以形成学情闭环；资源推送“千人一面”，无法适配个体认知差异；师生互动受时空限制，课堂参与度不足。尤其在“双减”政策背景下，学校对精细化教学管理、个性化学习支持的需求愈发迫切。与此同时，人工智能、大数据等技术日趋成熟，为教学模式的智能化升级提供了可能。本在线教育智慧教学平台旨在融合AI算法与教学业务逻辑，打通“教、学、评、管”全链路数据，构建自适应学习引擎与智能决策支持系统，助力教师精准施教、学生高效学习，推动教育从“经验驱动”向“数据驱动”转型，满足新时代智慧教育的落地需求。



## 三、定义

### 1、技术类

#### （1）OMO

OMO（Online\-Merge\-Offline）：线上线下融合教学模式，强调线上学习与线下教学场景的深度协同与数据互通。本平台支持OMO教学场景，实现线上课程学习、直播互动与线下作业、考试数据的全面打通。

#### （2）TRTC

TRTC（Tencent Real\-Time Communication）：腾讯实时音视频云服务，用于本平台直播推流、拉流及连麦互动，确保千人级并发直播场景下音视频延迟小于等于500ms。

#### （3）JWT

JWT（JSON Web Token）：一种开放标准（RFC 7519），用于在网络应用环境间安全地将信息作为JSON对象传输。本平台采用JWT实现用户身份认证与授权。

#### （4）WebSocket

WebSocket：一种在单个TCP连接上进行全双工通信的协议。本平台使用WebSocket实现白板指令同步、聊天消息、签到投票、作业批改通知等实时消息的推送与广播。

#### （5）RESTful API

RESTful API：一种基于HTTP协议、符合REST架构风格的API设计规范。本平台前后端数据交互采用RESTful API，通信数据格式为JSON。



### 2、业务类

#### （1）学情分析

学情分析：基于学员的学习行为数据（视频观看进度、作业完成情况、练习题得分等），通过算法计算知识点掌握度，生成可视化学习报告，帮助讲师和学员了解学习状态。

#### （2）知识点掌握度

知识点掌握度：衡量学员对某一知识点掌握程度的量化指标。本平台计算公式为：掌握度 = \(作业正确率 × 0\.5\) \+ \(练习题得分 × 0\.3\) \+ \(视频观看完成度 × 0\.2\)。

#### （3）OMO教学闭环

OMO教学闭环：从课程创建→选课学习→直播互动→作业批改→学情分析的完整教学流程，实现线上线下数据互通与教学效果量化评估。

# 第二部分 项目概述

近年来，OMO（线上线下融合）教学模式已成为教育行业数字化转型的重要方向。但是，大量传统在线教育平台在实时互动、数据闭环及多角色协同方面存在明显短板：高并发直播场景下音视频延迟高、卡顿严重；学习数据分散孤立，难以形成完整的学情画像；讲师、学员、运营管理人员三端协同效率低下，导致教学效果难以量化评估。

随着教育行业对智慧教学平台升级需求的日益迫切，以及《未成年人网络保护条例》《教育移动互联网应用程序备案管理办法》等政策对平台合规性的严格要求，构建一个集课程中心、直播课堂、智能学习、运营管理于一体的智慧教学平台已成为必然趋势。

本平台面向教育培训机构及学校师生开发，讲师可创建和管理课程、进行直播授课、发布与批改作业、查看学情分析；学员可选课学习、参与直播互动、提交作业、查看学习报告；运营管理员可进行用户管理、课程审核、数据监控、公告发布等操作。三端协同实现从课程创建到选课学习到直播互动到作业批改到学情分析的完整教学闭环。

该平台将提升教师备课效率40%，学生课程完课率提升至65%，家校沟通效率提升50%，帮助机构降低运营成本25%，最终实现教育机构的数字化转型与精细化运营。

### 2、后端逻辑架构设计

#### 2\.1 分层架构设计

1\.表现层：基于Vue\.js的前端架构，通过RESTful API与后端交互，使用Axios进行HTTP请求，WebSocket处理实时通信。

2\.业务逻辑层：Spring Boot 3\.2\+架构，封装核心业务逻辑，包括：

学情分析引擎：计算知识点掌握度、生成学习报告

直播管理服务：TRTC房间管理、信令转发

作业调度系统：自动批改、智能推荐

实时通信服务：WebSocket消息推送、白板指令同步

3\.数据服务层：基于MySQL 8\.0进行结构化数据存储，Redis 7\.0\+用于缓存高频数据（用户会话、课程信息、直播房间状态），支持读写分离和分库分表策略。

#### 2\.2 核心模块设计

系统采用企业级MVC模型，基于Spring Boot框架实现分层解耦：

Controller层：处理HTTP请求和WebSocket消息，参数校验，返回JSON响应

Service层：实现核心业务逻辑，事务管理，异常处理

Repository层：数据访问层，使用Spring Data JPA/MyBatis Plus操作数据库

DTO层：数据传输对象，前后端数据交换载体

Entity层：实体类，映射数据库表结构

## 三、推荐算法设计

### 3\.1 协同过滤推荐算法

基于学员错题行为的协同过滤推荐：

算法原理：

通过分析学员的错题记录，找出与目标学员具有相似错题模式的其他学员群体。若学员A和学员B在错题的题型、知识点分布等方面存在较高重合度，则认为这两个学员具有相似性。

1\.实现步骤：

2\.数据收集：获取所有学员的错题记录，按学员ID分组

相似度计算：

获取目标学员的错题ID集合

对于每个其他学员，计算其错题集合与目标学员错题集合的交集

相似度 = 共同错题数 / 并集错题数

3\.相似用户筛选：选择相似度最高的前10个学员

4\.题目推荐：

获取目标学员已做过的题目ID集合

遍历相似学员的错题记录，排除目标学员已做过的题目

按错题错误次数从多到少排序

每个相似学员最多推荐5道题

最终推荐不超过20道题目
